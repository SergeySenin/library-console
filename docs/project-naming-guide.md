# Guide по неймингу проекта library-console

## Формат заполнения

* Правила по конкретным классам.
* Фиксируются только итоговые решения.
* Кратко, без длинных аргументов и без примеров.

## Класс `Book`

### Роль класса

* Доменная сущность книги.
* Имя класса — короткое, предметное, в единственном числе.

### Нейминг полей

* Поля называют состояние книги без повторения имени класса.
* Текущий набор имён корректный:

    * `id`
    * `title`
    * `authorName`
    * `publicationYear`

### Нейминг параметров конструктора

* Параметры конструктора повторяют имена полей, если передают то же самое состояние.
* Текущий вариант корректный:

    * `id`
    * `title`
    * `authorName`
    * `publicationYear`

### Нейминг getter-методов

* Getter-методы повторяют имя поля в стандартном Java-формате.
* Текущий вариант корректный:

    * `getId()`
    * `getTitle()`
    * `getAuthorName()`
    * `getPublicationYear()`

### Нейминг приватных helper-методов

* Для внутренних проверок сущности используется префикс `validate...`.
* Текущий вариант корректный:

    * `validateId(...)`
    * `validateRequiredText(...)`

### Нейминг параметров helper-методов

* Параметры должны быть короткими и смысловыми.
* Текущий вариант корректный:

    * `id`
    * `textValue`
    * `fieldName`

### Язык сообщений

* Внутренние fail-fast сообщения в сущности пишутся на английском.

## Класс `ConsoleInputReader`

### Роль класса

* Консольный компонент чтения пользовательского ввода.
* Имя класса — предметное, отражает чтение данных из консоли.

### Нейминг полей

* Поле должно называть используемый консольный источник ввода без сокращений.
* Текущий вариант корректный:

    * `consoleScanner`

### Нейминг публичных методов

* Публичные методы должны называть действие чтения и его характер.
* Текущий набор имён корректный:

    * `readValidatedLine(...)`

### Нейминг параметров публичных методов

* Параметры должны отражать роль в процессе чтения и проверки ввода.
* Текущий вариант корректный:

    * `validator`
    * `validationErrorPrinter`

### Нейминг локальных переменных

* Локальные переменные должны отражать этап обработки пользовательского ввода.
* Текущий вариант корректный:

    * `enteredValue`
    * `validationResult`

### Правила именования для reader-класса

* Для методов чтения использовать префикс `read...`.
* Для параметров проверки использовать имя по роли, а не по конкретному сценарию.
* Для промежуточного ввода использовать имя, отражающее введённое пользователем значение.

### Связь нейминга с соседними классами

* Имена параметров должны быть согласованы с `ValidationResult` и validator-классами.
* Reader не должен содержать в нейминге предметные детали конкретного сценария книг.

### Язык сообщений

* Внутренние fail-fast сообщения в reader-классе пишутся на английском.
* Пользовательские сообщения reader не формулирует сам, а передаёт их через printer callback.

## Класс `ConsoleApplicationPrinter`

* Консольный компонент вывода общих сообщений приложения.
* Имя класса — предметное, отражает вывод на уровне приложения, а не на уровне книг.

### Нейминг публичных методов

* Публичные методы должны описывать печать конкретного вида сообщения или блока интерфейса.
* Текущий набор имён корректный:

    * `printApplicationHeader()`
    * `printMainMenu()`
    * `printApplicationFinishedMessage()`
    * `printValidationError(...)`
    * `printUnknownCommandMessage(...)`

### Нейминг параметров публичных методов

* Параметры должны отражать содержимое печатаемого сообщения.
* Текущий вариант корректный:

    * `errorMessage`
    * `userCommand`

### Нейминг приватных helper-методов

* Приватные helper-методы должны описывать печать повторяющегося фрагмента интерфейса.
* Текущий вариант корректный:

    * `printSeparatorLine()`

### Правила именования для printer-класса

* Для методов вывода использовать префикс `print...`.
* Имя метода должно указывать, что именно печатается: экран, меню, ошибка, результат, сообщение.
* В общих printer-классах не использовать предметные детали книжного сценария.

### Связь нейминга с соседними классами

* Имена методов должны быть согласованы с `ConsoleApplicationRunner` и `ConsoleCommandRouter`.
* Общий printer приложения не должен брать на себя вывод карточек книги и предметных результатов поиска.

## Класс `BookInputValidator`

### Роль класса

* Валидатор пользовательского ввода для книжных сценариев.
* Имя класса — предметное, отражает проверку входных данных уровня книг.

### Нейминг констант

* Константы называют ограничение и область его применения.
* Текущий набор имён корректный:

    * `MAX_TITLE_LENGTH`
    * `MAX_AUTHOR_NAME_LENGTH`
    * `MAX_TITLE_SEARCH_QUERY_LENGTH`
    * `MAX_AUTHOR_SEARCH_QUERY_LENGTH`
    * `MIN_PUBLICATION_YEAR`

### Нейминг публичных методов

* Публичные методы должны явно разделять валидацию и преобразование.
* Для проверки использовать префикс `validate...`.
* Для преобразования использовать префикс `parse...`.
* Текущий набор имён корректный:

    * `validateBookIdText(...)`
    * `parseBookId(...)`
    * `validateBookTitle(...)`
    * `validateAuthorName(...)`
    * `validatePublicationYearText(...)`
    * `parsePublicationYear(...)`
    * `validateTitleSearchQuery(...)`
    * `validateAuthorSearchQuery(...)`
    * `validatePublicationYearSearchQuery(...)`
    * `parsePublicationYearSearchQuery(...)`

### Нейминг параметров публичных методов

* Параметры должны отражать предметный ввод пользователя.
* Текущий вариант корректный:

    * `bookIdText`
    * `bookTitle`
    * `authorName`
    * `publicationYearText`
    * `titleSearchQuery`
    * `authorSearchQuery`
    * `publicationYearSearchQuery`

### Нейминг приватных helper-методов

* Приватные helper-методы должны называться по логике отрицательной проверки, если используются в условиях ошибки.
* Текущий набор имён корректный:

    * `containsNonDigit(...)`
    * `hasNoLetterOrDigit(...)`
    * `hasNoLetter(...)`
    * `startsOrEndsWithNonLetterOrDigit(...)`
    * `containsDisallowedAuthorCharacters(...)`

### Нейминг параметров helper-методов

* Для приватных helper-методов допустимо общее имя входного значения, если метод универсален.
* Текущий вариант корректный:

    * `value`

### Правила именования для validator-класса

* Для проверки использовать `validate...`.
* Для преобразования использовать `parse...`.
* Для текстового пользовательского ввода добавлять суффикс `Text`, если это важно для отличия от уже преобразованного значения.
* Для поисковых запросов использовать суффикс `SearchQuery`.
* Для отрицательных helper-методов допустима отрицательная семантика в имени, если она совпадает с использованием в `if`.

### Связь нейминга с соседними классами

* Имена validator-методов должны быть согласованы с `ConsoleBookHandler` и `ConsoleInputReader`.
* Validator должен использовать предметные имена книжных сценариев, а не общие абстрактные названия.

### Язык сообщений

* Сообщения пользовательской валидации в validator-классе пишутся на русском.
* Внутренние fail-fast сообщения о null-параметрах пишутся на английском.
