# Guide по неймингу проекта library-console

## Формат заполнения

* Правила фиксируются по конкретным классам и типам.
* Классы и типы идут по архитектурным слоям.
* Разделы внутри каждого типа идут по одному и тому же шаблону.
* Элементы внутри разделов идут в порядке появления в коде.
* Фиксируются только итоговые решения.
* Кратко, без длинных аргументов и без примеров.

## Класс `Book`

### Роль типа

* Доменная сущность книги.
* Имя класса — короткое, предметное, в единственном числе.

### Нейминг констант / полей

* Поля называют состояние книги без повторения имени класса.
* Текущий набор имён корректный:

  * `id`
  * `title`
  * `authorName`
  * `publicationYear`

### Нейминг конструктора и его параметров

* Параметры конструктора повторяют имена передаваемого состояния.
* Текущий вариант корректный:

  * `id`
  * `title`
  * `authorName`
  * `publicationYear`

### Нейминг публичных методов

* Публичные методы отражают чтение состояния сущности.
* Текущий набор имён корректный:

  * `getId()`
  * `getTitle()`
  * `getAuthorName()`
  * `getPublicationYear()`

### Нейминг параметров публичных методов

* В текущем типе публичные методы с параметрами отсутствуют.

### Нейминг приватных helper-методов

* Для внутренних проверок сущности используется префикс `validate...`.
* Текущий набор имён корректный:

  * `validateId(...)`
  * `validateRequiredText(...)`

### Нейминг параметров helper-методов

* Параметры должны быть короткими и смысловыми.
* Текущий вариант корректный:

  * `id`
  * `textValue`
  * `fieldName`

### Нейминг локальных переменных

* В текущем типе локальные переменные отсутствуют.

### Правила именования для типа

* Для доменной сущности использовать короткое предметное имя.
* Getter-методы должны повторять имя поля в стандартном Java-формате.
* Внутренние проверки состояния должны называться через `validate...`.

### Связь нейминга с соседними типами

* Имена полей и getter-методов должны быть согласованы с `BookCatalog`, `ConsoleBookPrinter` и `ConsoleBookHandler`.
* Доменная сущность не должна использовать нейминг консольного слоя.

### Язык сообщений

* Внутренние fail-fast сообщения пишутся на английском.

## Класс `BookInputValidator`

### Роль типа

* Валидатор пользовательского ввода для книжных сценариев.
* Имя класса — предметное, отражает проверку входных данных уровня книг.

### Нейминг констант / полей

* Константы называют ограничение и область его применения.
* Текущий набор имён корректный:

  * `MAX_TITLE_LENGTH`
  * `MAX_AUTHOR_NAME_LENGTH`
  * `MAX_TITLE_SEARCH_QUERY_LENGTH`
  * `MAX_AUTHOR_SEARCH_QUERY_LENGTH`
  * `MIN_PUBLICATION_YEAR`

### Нейминг конструктора и его параметров

* В текущем типе явный конструктор отсутствует.

### Нейминг публичных методов

* Публичные методы явно разделяют валидацию и преобразование.
* Для проверки используется префикс `validate...`.
* Для преобразования используется префикс `parse...`.
* Текущий набор имён корректный:

  * `validateBookIdText(...)`
  * `parseBookId(...)`
  * `validateBookTitle(...)`
  * `validateAuthorName(...)`
  * `validatePublicationYearText(...)`
  * `parsePublicationYear(...)`
  * `validateTitleSearchQuery(...)`
  * `validateAuthorSearchQuery(...)`
  * `validatePublicationYearSearchQuery(...)`
  * `parsePublicationYearSearchQuery(...)`

### Нейминг параметров публичных методов

* Параметры должны отражать предметный пользовательский ввод.
* Текущий вариант корректный:

  * `bookIdText`
  * `bookTitle`
  * `authorName`
  * `publicationYearText`
  * `titleSearchQuery`
  * `authorSearchQuery`
  * `publicationYearSearchQuery`

### Нейминг приватных helper-методов

* Приватные helper-методы называются по логике отрицательной проверки, если используются в условиях ошибки.
* Текущий набор имён корректный:

  * `containsNonDigit(...)`
  * `hasNoLetterOrDigit(...)`
  * `hasNoLetter(...)`
  * `startsOrEndsWithNonLetterOrDigit(...)`
  * `containsDisallowedAuthorCharacters(...)`

### Нейминг параметров helper-методов

* Для универсальных helper-методов допустимо общее имя входного значения.
* Текущий вариант корректный:

  * `value`

### Нейминг локальных переменных

* Локальные переменные должны отражать уже преобразованное или проверяемое значение.
* Текущий вариант корректный:

  * `bookId`
  * `publicationYearValue`
  * `currentYear`
  * `firstCharacter`
  * `lastCharacter`
  * `character`
  * `exception`

### Правила именования для типа

* Для проверки использовать `validate...`.
* Для преобразования использовать `parse...`.
* Для текстового пользовательского ввода добавлять суффикс `Text`, если это важно для отличия от уже преобразованного значения.
* Для поисковых запросов использовать суффикс `SearchQuery`.
* Для отрицательных helper-методов допустима отрицательная семантика в имени, если она совпадает с использованием в `if`.

### Связь нейминга с соседними типами

* Имена validator-методов должны быть согласованы с `ConsoleBookHandler` и `ConsoleInputReader`.
* Validator должен использовать предметные имена книжных сценариев, а не общие абстрактные названия.

### Язык сообщений

* Сообщения пользовательской валидации пишутся на русском.
* Внутренние fail-fast сообщения о null-параметрах пишутся на английском.
